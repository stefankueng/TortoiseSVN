Introduction

What is TortoiseSVN?

TortoiseSVN lets you work with files under Subversion version control directly from Windows Explorer. It is freely available under the GPL open source license. For more general information and screen shots, go to www.tortoisesvn.org.

Installation issues

Why don't the icon overlays appear?

  • If you are using Windows 95 the icon overlays won't appear. You can try the instructions for Windows NT4 below if you like, but it may still not work.
  • If you are using Windows NT4, you need to install the IE4 shell or desktop extensions to get a more recent version of Explorer. To do this install IE4, and choose Yes to install the active desktop. Don't worry, you can turn off the actual active desktop later by right clicking on it. It's the new version of Explorer that we are after.

    If you've already installed IE5, you must either:

    1. Uninstall IE5 and then install IE4 with desktop extensions, and then install IE5 again. What a palaver.
    2. Run IE5 setup with command line switches to install the IE4 desktop (shell) extensions. This may cause problems, read this usenet thread for the source of this information.

      The command must be run from the folder that contains the ie5setup.exe file. If the browser appears unstable afterwards just run the IE5 repair function.

      For Win95: ie5setup.exe /c:"ie5wzd /e:IE4Shell WIN /I:Y"

      For WinNT: ie5setup.exe /c:"ie5wzd /e:IE4Shell NTx86 /I:Y"

    For IE6:

    1. Run IE6 setup with command line switches to install the IE4 desktop (shell) extensions. The same caveats apply as for IE5.

      The command must be run from the folder that contains the ie6setup.exe file.

      For WinNT: ie6setup.exe /c:"ie6wzd /e:IE4Shell NTx86 /I:Y"

    The shortcut labelled Win NT Explorer on your start menu probably points to C:\WINNT\explorer.scf and doesn't get the overlays.

    Create a new shortcut to
    %windir%\Explorer.exe /n, /e
    and it may get the overlays.

  • If you're using an IntelliPoint mouse driver, and launching Explorer via a mouse click, you need to upgrade from version 3 to version 3.2 of IntelliPoint. Strangely, launching Explorer from the Start menu gives icons in this case, but not when launched from the mouse.

I followed the instructions, but there are still no icons?!?

You rebooted your PC off course?
So after installing the shell extensions you now have WinNT / Win95 and IE >= 5.5 installed?
Do a doubleclick on tortoiseproc.exe. If it complains about a missing msimg32.dll, try to find that dll on the web / another pc and copy it to your system.

Why do the overlay icons sometimes change to random graphics?

The Windows icon cache is a fairly buggy creature. You can solve this in one of the following ways:
  • Install Microsoft's TweakUI and run the option to rebuild icons.
  • Or increase the icon cache size. Go to HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Explorer and add a new String Value called "Max Cached Icons". The default value is 500 - try increasing it to 2048 (see http://support.microsoft.com/support/kb/articles/Q132/6/68.asp for more details).
  • Or delete the file called ShellIconCache in your Windows directory. And reboot.

Those icons are grotesque, I want better ones!

You can create your own icon set if you want. And if you do please contribute them to the rest of the world!

The icons should have at least two sizes (16x16 and 32x32 with 16 colors), for Windows XP there should also be a set with 48x48 pixels

to install the icons you should copy them to the directory where TortoiseSVN was installed. There you can either replace the existing icons or give them new names. If you choose not to overwrite the original ones then you also need to tweak some registry settings. Under HKLM\Software\TortoiseSVN\ change the keys ConflictIcon, InSubversionIcon and ModifiedIcon to the path and names of your own icon set.

after that you need to restart (or logoff) your computer so that explorer reads in the icons again. Alternatively you can flush the icon cache with the TweakUI tool from Microsoft. For WinXP you can get it here

The TSVN menu is missing when nothing is selected!
The TSVN menus are shown as ownerdrawn (no icons, no text) in some software

This is because the icons are completely ownerdrawn. The main menu is known to disappear under windows 98, ME, NT and 2000. Everything works fine under XP

I you don't like the way TSVN handles it's menu entries, you can do the following:
Just create a (DWORD) registry entry under: 

HKCU\Software\TortoiseSVN\OwnerdrawnMenus
and set it to 0
This will force TSVN to use the windows standard routine to draw menu entries. You'll have all the menu entries any time, but you will have ugly icons, since windows default only allows 10x10 icons. If you delete this registry key (the default) or set it to something else than 0 you have the nice icons back.

How do I uninstall TortoiseSVN?

Simply uninstall from Add/Remove Programs in the Windows control panel.

Daily use questions

I accidentally removed a file, and now it's gone!

You can do a revert on the parent folder where you deleted the file or directory. But use that carefully: you will loose all changes you made in that folder since you last committed!

When I remove a file it vanishes, how do I commit it?

Easy, you commit the whole directory! Right click in the Explorer window next to the file, and choose commit. I nearly always commit things from the top level directory - it is recursive through the whole project.

I always get an error when I check out a single file

There's nothing wrong with it. Subversion only allows to check out directories. A warning is not possible because TSVN has no possibility to check if a URL (e.g. http://svn.collab.net/svn/repos/trunk/README) is a file or a folder. But subversion itself complains if it's a file....

I'm in trouble renaming a file from uppercase to lowercase

Follow those steps:
  • Rename from "UPPERCASE" to "UPPERCASE_"
  • Commit the changes
  • Rename from "UPPERCASE_" to "lowercase"
  • Commit the changes
That way you won't loose the log-history of the file. Just remember to rename it "within" subversion, i.e. with the special rename command in the TSVN submenu. Just renaming in the explorer will corrupt your working copy.

I deleted a file and created a file with the same name. Now I can't commit?!?

Your're trying to fool subversion aren't you? Now seriously what did you expect to happen?
Assume you're in revision #55.
You removed a file using TSVN, which told it to mark the file for deletion with the next commit and to remove it from the explorer. It's not really gone from the revision yet. Then you created a file with the same name and added it to TSVN, which marks the file for adding with the next commit. It's not yet added to the repository. Now you commit. Poor subversion tries to decide whether to add/remove your file at the same time in revision #56. The commit fails.
You should have committed immediately after you deleted the file using TSVN's remove command.
You still have no clue what to do? Well, in that case:
  • Move that file to a safe place
  • Commit your changes. That tells the repository, that the file is gone in the first place
  • Copy the file back
  • Commit again

Where's the "Show Log" command?

It's available at two locations:
  • after an update there's a button to show the log for all the revisions you just updated to
  • select a file, right-click to get the context menu, click on "properties" to get the explorer properties dialog. Select the TortoiseSVN tab and then theres the button to show the log.

Can I mix and match with the Subversion command line client?

Yes, you can change from one client to another whenever you want.

Can I work on lots of working copies from different repositories at once?

Yes, you can. This is a standard feature of SVN. Each directory which was checked out of SVN remembers where it came from. You can even multiply select working copies which came from different places, and update or commit them all at once.

If you want to have different subdirectories to come from different repositories you can set a special Subversion property called svn:externals. Please look at the chapter External Definitions in the Subversion Book.

How does TortoiseSVN handle conflicts?

Imagine you check out revision 24 of a file and started editing it. In the meantime, someone else committed revision 25, and shortly after that revision 26. You tell TortoiseSVN to update your file now, and SVN will try to incorporate all the changes between revision 24 and 26 into your local file.

But if the changes made between 24 and 26 were too close to or even overlap with the changes you have made, SVN detects a conflict and creates conflict markers in the file. Subversion also creates three special files for every conflicted file:

  1. filename.*.mine This is your file as it existed in your working copy before you updated your working copy - that is, without conflict markers. This file has your latest changes in it and nothing else.
  2. filename.*.rOLDREV This is the file that was the BASE revision before you updated your working copy. That is, the file that you checked out before you made your latest edits.
  3. filename.*.rNEWREVThis is the file that your Subversion client just received from the server when you updated your working copy. This file corresponds to the HEAD revision of the repository.
At this point TortoiseSVN will not allow you to committ the conflicted file until you resolved the conflict. To tell TortoiseSVN that you resolved the conflict simply delete the three additional files or use the command resolve to do that for you.

Misc

Does TortoiseSVN support SSH?

TSVN uses subversion. And subversion supports ssh either via https (repository reachable via apache) or svn+ssh:// (repository reachable via svnserve).
Please have a look at the subversion book and the INSTALL file of subversion on how to use svn+ssh://

Can I lose data, when I use TortoiseSVN?

TSVN uses Subversion. Nuff said :-)
Seriously not, read the Subversion pages. You can screw up your working copy any way you like, but if you commit your data regularly, you can always revert to a functional state. TSVN was not made to protect you against stupidity (e.g. Format C:).

There is however one issue involved. TSVN is always built against the latest subversion release. If the subversion team decides to change the database layout, then your TSVN client might not match your SVN repository and cause havoc. In this case you have to migrate the repository first. Check the subversion book on how to do this.
TSVN currently has no way to determine the version of the repository it is accessing. This may become a feature of a post 1.0 subversion release.
